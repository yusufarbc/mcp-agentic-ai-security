What is the Model Context Protocol (MCP)?

You can think of the Model Context Protocol (MCP) as an "App Store" or a universal "USB-C port" for AI. Just as the App Store is a standard platform that allows you to easily add new features (apps) to your phone, MCP simplifies adding new capabilities (tools) to AI models. It grows the ecosystem, enhances security, and paves the way for innovation with a "write once, run anywhere" logic.

In Practice

  • Complex World (Without MCP): Every AI (LLM) needs a special "adapter" to connect to a different tool (weather API, calculator, database, etc.). This is like carrying hundreds of different chargers; inefficient, complex, and unmanageable.
  • Simple World (With MCP): MCP offers a universal standard for all tools and models. When a tool is MCP-compliant, it can be instantly used by all MCP-supported AIs.

Technical Operation

The core flow of MCP occurs between a client (usually the LLM itself or an agent) and an MCP server (hosting the tool). This process consists of a few simple steps:

  1. Discovery: The client asks what tools are available on the server and what functions (e.g., `getCurrentWeather`) with which parameters (`location`) these tools offer.
  2. Request: When the LLM decides to use a specific tool, the client sends a request to the server in a standard JSON format to invoke that tool's function.
  3. Execution: The MCP server receives the incoming request, validates it, and runs the code of the relevant tool.
  4. Response: The result obtained after the tool runs (e.g., weather information) is sent back to the client in a standard JSON format. The LLM uses this output to generate a response for the user.

Brief Summary of the Research

The paper systematically puts the MCP server lifecycle (creation, deployment, operation, maintenance) and security risks into a table.

  • Architectural Boundaries: The Host/Client side manages tool discovery, while JSON-RPC servers provide resources and tools; transport layer choices (STDIO vs HTTPS/SSE) and authentication (OAuth, mTLS, scoped token) determine the attack surface.
  • Four Actors, 16 Scenarios: Developer-originated (shadow server, namespace collision), external attacker (indirect prompt injection, setup fraud), malicious user (STAC chain, sandbox escape, session reuse), and software/configuration errors (credential leak, command injection).
  • Empirical Data: A scan of 1,899 servers found 7.2% general vulnerabilities, 5.5% tool poisoning risk, 66% code smells, and 14.4% error patterns; these findings are missed without MCP-specific scanners.
  • Benchmark Results: MCPGAUGE does not provide benefit in all scenarios; < 60% success was observed in MCP-Universe & LiveMCP-101; MCPToolBench++ highlights context diversity bottlenecks; AutoMalTool can bypass defenses, but MCP-Guard achieves 96% detection accuracy.
  • Defense Strategies: IFC + taint tracking, deployment-specific sandbox profiles, TLS/mTLS + OAuth 2.1 resource indicators, short-lived scoped tokens, plan-based testing + logging + red teaming, signed packages/SBOM, and automated server generation from OpenAPI.

Security and Threat Landscape

Threat Taxonomy

  • 4 actors (malicious developer, malicious user, external attacker, software errors)
  • 16 scenarios: tool poisoning, prompt injection, input validation errors
  • 7.2% vulnerabilities, 5.5% tool poisoning, 66% code smells in 1899 server analysis
  • Benchmark examples: MCPGAUGE, MCPToolBench++, LiveMCP-101
  • Red team output: AutoMalTool can bypass most defenses; MCP-Guard reaches 96% accuracy

Defense Layers

  • IFC + taint-tracking, least privilege, sandbox profiles
  • TLS/mTLS, OAuth 2.1 + scoped token, SBOM and signed package
  • Guard models, AutoMalTool red team procedures
  • AutoMCP compiler from OpenAPI, plan-based tests
  • MCP scanners for CI/CD, human-approved guard flows, short-lived credentials

Layered controls are recommended against Prompt injection, data poisoning, DoS, and privacy violations; reference papers detail FPETS/FHE, rate limiting, and guard certificates.
